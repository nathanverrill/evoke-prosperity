Brightspace Integration — Security Model & Setup

Verified against the live tenant (charge.yacenter.org, "EVOKE Test") July 19–21, 2026
01 · Security model

OAuth only, two token paths, least privilege by construction

Everything that runs today goes through Brightspace's OAuth 2.0 (Manage Extensibility app registration). There are no shared service passwords, no API keys in student paths, and no LTI launch in use. Two distinct token paths, deliberately kept separate:

Per student The student's own token Minted fresh at every real login. Used for evidence submission and grade sync, so every submission is attributed to the right person in the gradebook — and a student's token can only ever see what that student can see. No shared credential to leak or over-scope.
Admin session Service-connection token Minted by the admin's one-click "Connect Brightspace" in /admin, cached in process memory only — never written to disk or database. Used for read-only pulls (assignments, classlist, grades). Expires with the token or a restart; reconnecting is one click.

Boundaries that hold by design

  • Two identity systems that never mix. Real students and instructors are Brightspace accounts underneath (OAuth). The EVOKE admin is a local bcrypt login with no Brightspace identity behind it — signing into /admin never touches Brightspace, and the admin's connect flow uses its own state cookie so it can't collide with a learner login in the same browser.
  • CSRF-protected OAuth flows. Random per-flow state values in origin-bound cookies, validated at the callback; the registered redirect URI must match exactly.
  • Reads are free, writes are deliberate. Pulling assignments is read-only and safe to repeat. Linking an assignment to a mission is an explicit admin action with database-enforced 1:1 uniqueness in both directions — an assignment already linked elsewhere 409s by name rather than silently re-linking.
  • EVOKE never writes rosters, groups, or enrollment. Brightspace Groups are the read-only source of truth for teams; the old manual roster import was deliberately removed. The district's enrollment process stays authoritative.
  • Submissions never block on Brightspace. Evidence stores locally first; a background worker pushes it to the Dropbox Folder using the student's own token. If Brightspace is down, the student's submission still exists and is marked for the sync-failed path — nothing is lost.

Not in use, by choice: LTI 1.3 launch (an alternate login path, unwired) and the legacy APP_KEY/APP_SECRET ID-Key service account (dead code — do not set one up expecting it to be required).

02 · Brightspace setup

One-time setup, per course offering

Needs someone with instructor/course-admin rights, once:

  1. Register the OAuth app (Manage Extensibility). Yields BRIGHTSPACE_CLIENT_ID / BRIGHTSPACE_CLIENT_SECRET. The registered Redirect URI must match BRIGHTSPACE_OAUTH_REDIRECT_URI character-for-character — a mismatch fails OAuth state validation. Scope: core:*:*.
  2. Create the course offering the normal way; its org unit ID becomes BRIGHTSPACE_ORG_UNIT_ID.
  3. Create a Group Category with named groups (e.g. Alpha, Bravo) and enroll students into them. EVOKE reads these at every login to assign teams — it never creates or edits groups.
  4. Enroll learners normally (SIS, instructor add — whatever the institution does). EVOKE has no roster import; accounts are provisioned automatically at first OAuth login.
  5. Click "Connect Brightspace" in /admin to mint the service-connection token the pull/read actions use. Repeat after a restart or token expiry.

Per mission — repeat for each one going live

  1. Create the Dropbox Folder in Brightspace — and publish it. Creating it is not enough: set its Availability/Visibility so students can actually see it.
  2. Pull → Link in /admin. Pull fetches and validates live assignments (read-only); Link attaches one assignment ID to one EVOKE mission.
  3. Release the mission in EVOKE (released_at in the mission editor) — EVOKE's own gate, independent of Brightspace's publish state. Both must be true before the mission genuinely works.
The one silent failure mode, confirmed live: an assignment can exist, be correctly mapped, and be visible to the admin token — yet 404 every student submission because it was never published to students. Brightspace gives no other signal. If a submission shows brightspace_sync_failed with a correct mapping, check the folder's Availability/Visibility before suspecting code.
03 · What runs itself

Automatic once setup is done

FlowHowTrigger
Player account creation Provisioned on first OAuth login, linked to the Brightspace user ID Every login
Team assignment Read from Brightspace Groups — a live sync, not a one-time import Every login
Evidence → Brightspace dropbox Event-driven background worker, student's own token; status tracked submitted → brightspace → graded On submit
Grade pull Admin clicks per mission; no Brightspace-side step needed On demand

Failed pushes are logged and the local copy kept; there is no automatic retry today. A grade webhook (Brightspace pushing grades to EVOKE) is possible but unconfigured — on-demand pull covers the pilot.

04 · Quick diagnostic

"The student submitted but it's not in Brightspace"

Compare the assignment list as seen by the service token versus that student's own token. If the folder is missing from the student's-eye view, it's the publish step above — someone with instructor rights needs to set the folder's Availability/Visibility. Mapping, tokens, and EVOKE's release gate can all be correct and this still blocks every submission.
