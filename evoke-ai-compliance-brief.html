AI Guardrails & Data Handling — Technical Brief

Verified against the live stack July 21, 2026 · Not legal advice — companion to SAFETY.md and COMPLIANCE_TODO.md
AI may auto-reject. Only a human can make something visible. The single operating rule every control below implements. AI cuts review volume; it is never the last check on anything a learner or classmate sees.
01 · The AI chain

Every model call stays inside our own containers

All four AI call sites — B1llbot chat, the AI Coach worker, the Field Report worker, and the Minecraft lore bridge — route through one self-hosted gateway. No student text reaches a third-party vendor: the “no training on student data” claim is a property of the architecture, not a contractual promise.

App call sites main.py · workers.py · bridge.py — user messages only; no client-side prompts
→
LiteLLM gateway Pre-call: PII mask + content filter. Post-call: same two on the reply. Default-on for every caller.
→
OpenWebUI Injects the B1llbot system prompt server-side — callers cannot omit or replace it
→
Ollama Open-weight model (qwen3:8b), local inference, nothing leaves the box
Entire chain runs on one Docker network operated by this project. Swapping the backend for a hosted provider is a gateway config change — every guardrail stays in place — but is gated on a written no-training DPA and named-vendor disclosure to the district (SAFETY.md §3).
02 · Gateway guardrails

What runs on every request, verified live

ControlDirectionBehaviorStatus
PII masking (Presidio) In + out Names, emails, phone numbers, locations, card numbers masked before the model sees them and again on the reply. SSNs are blocked outright. Live · tested
Content filter In + out Violence, weapons, and prompt-injection/jailbreak patterns blocked pre-call; violence filter re-run on output. Live · tested
Crisis language pass-through In Self-harm/crisis text is deliberately not blocked — blocking it would silence B1llbot’s crisis redirect (verified regression, since fixed). It must reach the model so the redirect can fire. By design
Content-safety model (Llama Guard 3) In + out Model-based classification for harm categories, still fully local (same Ollama container). Planned · Ph 2
Crisis escalation routing In Flag routed to a named human faster than the ordinary queue — the pipe exists in design; the queue it routes to is unbuilt. Planned · Ph 3
Rate limits & budgets — Per-caller ceilings so a runaway session hits a limit, not a bill. Planned · Ph 4

“Tested” = exercised against the running gateway on July 21, 2026: jailbreak input blocked with a specific category match; name/phone masked end-to-end through a real B1llbot reply; SSN blocked pre-call.

03 · Defense in depth

Three layers, each doing what the others can’t

Behavior
System prompt (server-side) Persona limits, age-appropriate language, no grading, no real-world financial advice, and the crisis redirect (drop character, point to a trusted adult). Injected by OpenWebUI, never supplied by the client. Steerable by a determined attacker — which is why it is never the enforcement layer.
Enforcement
Gateway (LiteLLM) The controls in §02, applied to every call site by construction — a new AI feature gets guardrails by pointing at the gateway, not by remembering to add them.
Judgment
Human facilitator Anything visible to other learners requires facilitator approval; safety-critical flags route to a named person on a real response window. AI never approves — it only rejects or queues.
04 · Content handling

How learner content is treated internally

Photo evidence (Field Tablet). The raw photo is never written to storage — by design, after finding the original design cached raw JPEGs carrying phone EXIF including GPS.

  1. Photo arrives and exists in memory only, for exactly as long as rendering takes.
  2. EXIF orientation is applied to the pixels (so portrait shots render upright), then discarded with the rest of the metadata.
  3. Pixels are re-encoded into a styled Field Evidence PDF — re-encoding drops the entire EXIF block, including GPS, by construction rather than by a stripping step that could be forgotten.
  4. The PDF is the only stored artifact. There is no raw photo to leak, cache, or forget to delete.

Chat text. Masked by the gateway before the model sees it and again on the reply. Chat events are the planned source for the facilitator moderation queue and the retention window.

Storage locality. All persistence is self-hosted in-stack — Postgres, MinIO object storage, OpenSearch, Redpanda — with per-cohort instance isolation. No behavioral advertising, no data sale, no cross-cohort sharing, enforced by architecture, not just policy.

Not yet aligned to the evidence-pipeline standard:

  • Avatar uploads and Minecraft quest screenshots still store the raw uploaded bytes, EXIF included — the P1 “strip EXIF before storage” item. The Field Evidence pipeline is the model to converge on. Gap
  • The app’s own B1llbot chat log stores the raw message before gateway masking, with no retention window or deletion path. Gap

06 · Hardening before pilot

Dev-shaped edges that must close first

  • Gateway keys default to a dev secret and port 4000 is published to the host — rotate keys, keep the gateway network-internal. Gap
  • Surface guardrail rejections to the learner with the specific reason (“looks like it includes a phone number”) instead of a generic error — required by SAFETY.md §4, not yet implemented in the app layer. Gap
  • Post-call PII masking currently masks B1llbot’s own fictional people and places; scope the output pass to high-risk entities (phone/email/SSN/card). Tuning
  • Add a chain smoke test — a rotated upstream key silently took the whole AI chain down once; a one-line health probe catches that in minutes, not at review time. Gap
