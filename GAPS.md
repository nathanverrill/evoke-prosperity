# GAPS.md — what this build is still missing

An honest audit of the gamified transmedia experience as specified in [`BUILD_PLAN.md`](BUILD_PLAN.md) + [`UI_SPEC.md`](UI_SPEC.md) + [`GAME_DESIGN.md`](GAME_DESIGN.md), against the ambition in `docs/canon/` and the Urgent Evoke lineage. Five lenses: technical, narrative, UX, gamification, business & adoption. Each item is tagged **[decision]** (needs Nathan/stakeholder call before anyone builds) or **[build]** (scoped work), with the ones that can sink the pilot called out first.

---

## The five that matter most

1. **~~Minecraft Java vs. Bedrock~~ [RESOLVED → build risk]** — decided: the server is Java (Fabric) modified to accept Bedrock clients via **Geyser + Floodgate**, per `BUILD_PLAN.md`. The residual risks to engineer around: Floodgate players have prefixed usernames and non-Mojang UUIDs (the reward bridge's username matching and the account-linking flow must handle both identity shapes); Geyser is version-locked to specific Minecraft/Fabric builds (pin everything, upgrade deliberately); and some Java-side visuals/interactions degrade on Bedrock clients — test the actual quest locations in `true_oasis` from a Bedrock device, not just a Java client.
2. **~~No social layer~~ [PARTIALLY RESOLVED → build]** — a class-wide activity feed now exists (`GET /api/activity`, a new ACTIVITY WORKER in `workers.py`), showing every learner's `AwardGranted`/`QuestCompleted` events with display names and mission/quest titles, rendered live on the Operations Hub. Verified end-to-end with a real stack and a headless-browser screenshot. **Still missing:** the deeper peer-*interaction* layer this item originally called for — a gallery of completed evokations, peer reactions/comments on debrief pages, `InsightPublished(source=peer)` isn't emitted by anything yet. The feed makes the cohort's activity *visible*; it doesn't yet let learners *react* to each other's work.
3. **No revise-and-resubmit loop [build]** — the pipeline is one-way: submit → AI → teacher → done. PBL pedagogy (and plain fairness) needs "needs revision → improve → resubmit" as a first-class path: a `RevisionRequested` event, a resubmission state in the timeline, and award logic that rewards improvement rather than punishing the first draft. Right now a weak first submission is a dead end.
4. **Graphic-novel content pipeline is undefined [decision + build]** — `UI_SPEC.md` says panels load from `static/content/`, but nothing defines how many chapters exist, who produces the art, what format/resolution, or the chapter↔mission unlock map. The transmedia promise dies without this. Needed: a `chapters.json` manifest schema (panels, captions, unlock conditions), placeholder chapters for all 6 weeks, and an authoring convention the narrative team can fill without engineering help.
5. **Collective world-state is missing [decision → build]** — the story is about *Keel recovering*. Nothing in the product makes the class's combined effort visible: no town-repair meter, no shared map that improves as missions complete, no "the water is flowing again" moment when the cohort hits a milestone. This is the single strongest transmedia glue available — the same `MissionCompleted` events could drive a class-wide progress visual on the Hub *and* physical changes in the Minecraft world (RCON `setblock`/`fill` region swaps). It's also the piece that makes Minecraft and the web app feel like one world instead of two products.

## Business & adoption

Missing lens in earlier passes of this doc — these are less "will the pilot work" and more "will anyone beyond one classroom ever run this," which matters because `ARCHITECTURE.md` is explicitly scoped to a single technical operator.

| Gap | Why it matters | Tag |
|---|---|---|
| **~~No operating model for a non-technical school~~ [RESOLVED → build items below]** — decided: one AWS instance per cohort, spun up from a golden AMI, operated by EVOKE (not the school). The school needs zero infrastructure competence; this is a natural concretization of `ARCHITECTURE.md`'s existing "one box per org" principle. Mitigates cross-tenant data leakage, blast radius of a security/moderation incident, and gives a clean data-ownership answer (a district's data lives on its own dedicated instance, exportable/deletable as a unit). **Does not mitigate:** content safety within a cohort's own B1llbot chat, griefing/harassment between students in the same shared world, or whether that cohort's AI calls reach a third-party vendor — those remain per-instance problems, just contained to one cohort instead of all | Resolves the "who runs this" question; residual work is fleet operations, not architecture | [RESOLVED → build] |
| **AMI must not bake in shared secrets** — `.env.example`'s placeholder values (`RCON_PASSWORD`, `DATABASE_URL`, `MINIO_SECRET_KEY`, etc.) must not ship identical across every instance cloned from the same AMI, or the per-cohort isolation story is cosmetic — one leaked secret compromises every cohort at once | Needs a first-boot (cloud-init) step that generates unique secrets per instance before anything starts | [build] |
| **No fleet provisioning tooling** — spinning up a new cohort instance needs to be IaC-driven (even a simple AWS SDK script), not click-ops, including DNS/TLS automation and the roster/LTI credential handoff from the district. **Domain scheme decided:** `{cohort-slug}.app.<root-domain>` (web app) and `{cohort-slug}.mc.<root-domain>` (Minecraft), one apex domain EVOKE owns (not per-district custom domains — that reintroduces the "school needs DNS competence" problem). Route53 record updated per instance (not hardcoded), so a cohort's domain survives instance replacement/patch rollout. One wildcard TLS cert per subdomain zone, terminated per-instance (Caddy), not behind a shared fleet-wide load balancer — consistent with `ARCHITECTURE.md`'s "one box does everything," avoids reintroducing a shared failure point across cohorts. **Root domain itself is still an open decision** — nothing owned/chosen yet | Click-ops doesn't scale past a handful of cohorts and is where mistakes (like the secrets issue above) get made. Also a hard sequencing dependency: LTI/Brightspace registration needs the domain live (DNS+TLS resolving) *before* a district admin can register launch/JWKS/redirect URLs, so domain assignment has to be step one of provisioning, not an afterthought | [decision + build] |
| **No single-doorway entry point — deferred on purpose, not forgotten** — considered a shared `evokeprosperity.com` front door (LTI/direct visitors land there, get associated with "their world") instead of visitors always knowing their cohort's subdomain directly. A doorway only helps the secondary case of a cold/bookmarked visit without an LTI launch; revisit with **(a)** once enough cohorts exist that "which subdomain was mine" becomes an actual support complaint, and don't build **(b)** at all — it undoes the isolation the AMI-per-cohort model was chosen for | Avoids solving a problem that doesn't exist yet at the cost of the isolation property just decided above | [decision — deferred] |
| **No fleet-wide monitoring view** — `config/prometheus.yml`/`config/alerts.yml` exist per-instance from the July sprint, but nothing aggregates health across N running cohort instances into one view | Can't staff on-call effectively if "healthy" requires logging into every box individually | [build] |
| **No moderation escalation SLA** — even with AI first-pass triage (chat/image classifiers), someone has to receive and act on a flagged item within a defined time; nothing specifies who, or how fast | Any escalation touching a minor's safety carries real stakes and, in serious cases, mandatory-reporting obligations that can't be delegated to AI | [decision] |
| **No fleet-wide patch/version rollout mechanism** — a golden AMI is good for isolation, bad for pushing a bug fix or new mission to N already-running instances without one | Needed before running more than one or two cohorts at once, or every fix becomes a manual per-instance operation | [build] |
| **No offboarding runbook** — end of semester/contract needs a defined snapshot-and-terminate or hard-delete path, not improvisation, especially for FERPA deletion requests | The data-ownership pitch only holds if deletion is actually a clean, provable action | [build] |
| **No standards alignment or gradebook translation** — missions carry ad hoc codes (`CES 5.1`, `FL.HS.5.1`) but nothing maps the campaign to an actual state financial-literacy graduation requirement, and AI/instructor feedback never resolves into a number a teacher can put in a gradebook | This is what a district procures against; without it, EVOKE Prosperity is enrichment, not curriculum, in a purchasing conversation | [decision] |
| **No efficacy/research validation plan** — the Superpower framework this campaign is built on (Freeman & Hawkins 2016, `GAME_DESIGN.md` §4) is literally the first of a three-paper World Bank series whose second installment is an experimental research study on a real pilot (Colombia, human trafficking/displacement challenge). EVOKE Prosperity has rubrics but no equivalent plan — no pre/post skill assessment, no control comparison, no way to demonstrate the financial-literacy claim actually landed | Needed before anyone pitches this with an efficacy claim, and ironic to skip given the source framework's own rigor | [decision] |

### AI-augmentable vs. irreducibly human support needs

The AMI-per-cohort model shifts "who operates this" from the school to EVOKE, which makes fleet support a real, ongoing cost — sized to cohort count, not eliminated by architecture. AI compresses *volume* and *latency* on known patterns; it does not remove the need for a human wherever a call is high-stakes, novel, or touches a minor's safety or a legal obligation.

| Support need | What AI can compress | What stays human |
|---|---|---|
| Provisioning | Generating the IaC; once built, spin-up itself can be scripted (automation, not really AI-in-the-loop) | Approving new-cohort provisioning; handling exceptions (custom domain, district SSO/firewall) |
| Monitoring/alerting | First-pass triage against `config/alerts.yml`; auto-remediation of known patterns (restart a crashed container, clear disk) | Novel failures; judgment calls with real stakes (e.g. restoring a corrupted world from backup); staffing on-call hours at all |
| Live-class incident response | A support agent that self-diagnoses and fixes common "won't load" issues in real time | Severe/novel issues, and a fast, clear escalation path when the AI can't fix it — time-sensitive because it's happening during a live 45-minute period |
| Moderation (B1llbot transcripts, Minecraft chat, screenshot uploads) | Classifiers doing first-pass flagging (toxicity, image moderation, jailbreak-attempt detection) — cuts review volume by orders of magnitude | **Cannot be delegated:** anything touching a minor's safety needs human judgment and, in serious cases, carries mandatory-reporting obligations |
| Teacher/customer support | A RAG-based bot over the product's own docs handles most "how do I" tickets instantly | Account-level actions needing authorization (roster fixes, refunds, deletion requests); early customers who want a human relationship while trust is being built |
| Fleet patch rollout | Generating the rollout diff/script; pre-flight checks per instance | The decision to push a change live across running cohorts mid-semester, and any rollback call |
| Offboarding | The runbook itself, scripted and AI-executed given approval | The sign-off to actually delete a district's data — legal/compliance stakes |
